## ADR-007: SSAS данные собирать напрямую через AdomdClient

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Данные о структуре SSAS-кубов уже собираются CubeMetaData.exe в `lineage.CubesStructure`.
Можно либо читать оттуда, либо подключаться к SSAS напрямую.
Чтение из существующей таблицы проще, но зависит от внешнего pipeline и не даёт контроля
над частотой и полнотой обновления.

### Решение
SsasExtractor подключается напрямую к SSAS через `Microsoft.AnalysisServices.AdomdClient.NetCore.retail.amd64`.
Собирает Server → Database → Cube → DSV → Named Queries/Tables → Columns, Dimensions → Attributes,
MeasureGroups → Measures. Named Query SQL парсится через TsqlLineageParser.
Данные записываются инкрементально в `lineage_v2`.

### Отвергнутые альтернативы
- **Читать из lineage.CubesStructure** — зависимость от CubeMetaData pipeline, нет контроля над обновлением, данные могут быть устаревшими

### Последствия
(+) Полный контроль над сбором данных, независимость от legacy pipeline
(+) Можно парсить DSV Named Queries напрямую для column-level lineage
(–) Нужен NuGet `Microsoft.AnalysisServices.AdomdClient.NetCore.retail.amd64`
(–) Нужен сетевой доступ к OLAP-VDI серверам

## ADR-006: SSIS пакеты загружать из TFS, а не с диска

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Legacy CubeMetaData читает .dtsx файлы с локального диска (`settings.json → PathToSSISPackages`).
Это ненадёжно — на диске могут быть устаревшие версии, не совпадающие с тем, что реально
развёрнуто. TFS (http://tfs-tsum:8080/tfs/tfs_olapcollection) содержит актуальные версии.

### Решение
TfsClient скачивает .dtsx файлы через TFS REST API v7.0 с Windows Integrated Auth.
Используется endpoint `_apis/tfvc/items?path=...&api-version=7.0`.
Пакеты скачиваются в память (не на диск), парсятся и результат записывается в граф.

### Отвергнутые альтернативы
- **Читать с локального диска** — могут быть устаревшие версии, зависимость от ручного копирования
- **GitHub repo bascomo/SSIS** — содержит пакеты, но не является source of truth для production

### Последствия
(+) Всегда актуальные версии пакетов
(+) Не нужно ручное копирование файлов на сервер
(–) Зависимость от сетевой доступности TFS
(–) Нужна Windows Integrated Auth (Kerberos/NTLM) для доступа

## ADR-005: Отдельная таблица NodeHistory для аудита изменений

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Граф lineage меняется со временем: добавляются новые объекты, удаляются старые,
меняются связи. Для анализа эволюции архитектуры и потоков данных нужна история.

### Решение
Отдельная таблица `lineage_v2.NodeHistory` с полями: NodeId, ChangeType (Created/Updated/Deleted/Restored),
ChangedAt, ChangedByRunId, OldValues JSON, NewValues JSON. Заполняется при каждом MERGE.

### Отвергнутые альтернативы
- **Только CreatedAt/UpdatedAt/IsDeleted на узлах** — нет истории промежуточных изменений, нельзя анализировать эволюцию
- **Temporal Tables (SQL Server)** — системные temporal tables менее гибки для кастомного аудита

### Последствия
(+) Полная история: когда узел появился, когда менялся, когда исчез
(+) Можно анализировать эволюцию архитектуры DWH
(–) Таблица будет расти (нужна стратегия ретенции)

## ADR-004: MechanismNodeId — FK на Node вместо строкового поля

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Каждое ребро графа создаётся каким-то механизмом (view, SP, SSIS-пакет).
Нужно хранить ссылку на этот механизм. Варианты: строковое поле MechanismLocation
или FK на таблицу Node (поскольку механизм — тоже узел графа).

### Решение
`Edge.MechanismNodeId INT NULL FK → Node`. Механизм — тоже узел графа.
Это позволяет навигировать "какие связи создаёт этот view?" и "через какие механизмы
данные попадают в эту колонку?".
Unique constraint: `(SourceNodeId, TargetNodeId, EdgeType, MechanismNodeId)`.

### Отвергнутые альтернативы
- **Строковое поле MechanismLocation** — нет ссылочной целостности, нельзя навигировать по механизмам

### Последствия
(+) Ссылочная целостность, навигация по механизмам
(+) Unique constraint защищает от дублей при зеркальных серверах
(–) Нужно сначала создать узел-механизм, потом ребро (порядок вставки)

## ADR-003: FQN через точку как natural key, без ParentNodeId

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Каждый узел графа должен быть уникально идентифицирован. Нужен natural key
и способ навигации по иерархии (Server → DB → Schema → Table → Column).

### Решение
`FullyQualifiedName NVARCHAR(1000) UNIQUE` — разделитель точка.
Примеры: `DWH-VDI.DWH.dbo.FactSales.Amount`, `OLAP-VDI.MOVofGoodsDynamic.SalesCube.DSV.nq_FactSales`.
Иерархия определяется из FQN + справочника NodeType. ParentNodeId не хранится — вычисляется.

### Отвергнутые альтернативы
- **ParentNodeId FK** — дополнительное поле, требует поддержки при MERGE, избыточно при наличии FQN
- **Разделитель бэкслеш (\)** — менее привычен для SQL-контекста, визуально не принципиально

### Последствия
(+) Простая модель, нет лишних FK
(+) FQN самодостаточен для определения иерархии
(–) Парсинг FQN для навигации по иерархии (незначительный overhead)

## ADR-002: Иерархический справочник NodeType

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Узлы графа принадлежат разным платформам (SQL Server, SSAS, SSIS, mETL) и имеют
разные уровни иерархии (Server → DB → Table → Column vs Server → Cube → MeasureGroup → Measure).
Нужен способ классифицировать узлы с учётом платформы и уровня вложенности.

### Решение
Таблица `lineage_v2.NodeType` с самоссылкой: `NodeTypeId, Name, ParentTypeId FK, Level`.
Каждый Node ссылается на NodeTypeId (лист иерархии).
Пример: Measure (Level 5) → MeasureGroup (4) → Cube (3) → SSAS Database (2) → SSAS Server (1) → SSAS Platform (0).

### Отвергнутые альтернативы
- **Строковое поле NodeType + отдельное PlatformName** — нет иерархии, нет связи между типом и его контекстом
- **Enum в коде** — не расширяемо без перекомпиляции

### Последствия
(+) Единая иерархическая классификация для всех платформ
(+) Расширяемость — новые типы добавляются без изменения кода
(+) Навигация: зная тип узла, можно определить ожидаемую структуру FQN
(–) Дополнительная таблица и JOIN при запросах

## ADR-001: Эволюционная миграция из CubeMetaData, схема lineage_v2 в MetaMart

**Дата:** 2026-03-28
**Статус:** Принято

### Контекст
Существует работающее решение CubeMetaData (.NET Framework 4.7.2) с SQL Parser,
SSIS Parser, SSAS Collector. Данные хранятся в `MetaMart.lineage.*` (2.4M объектов, 194K связей).
Нужно решить: переписывать с нуля или эволюционировать.

### Решение
Эволюционная миграция: портировать логику из CubeMetaData в новый .NET 8 solution LineageBuilder.
Новые данные записывать в схему `lineage_v2` в той же базе MetaMart на DWH-VDI.
Legacy схема `lineage` остаётся нетронутой, оба решения работают параллельно.

### Отвергнутые альтернативы
- **Переписывание с нуля** — потеря накопленных edge-case'ов в Visitor.cs (1944 строки), высокий риск регрессий
- **Расширение legacy lineage** — сломает текущий pipeline (CubeMetaData + uspGenerateIdentifiers + LinkUpd)
- **Отдельная БД** — лишняя зависимость, данные далеко от источника

### Последствия
(+) Существующий pipeline продолжает работать
(+) Можно сравнивать результаты legacy vs новый граф для валидации
(+) Постепенная миграция без big bang
(–) Две схемы в одной БД, нужна документация что где
